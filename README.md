# Portfolio Website

This repository contains the codebase for my personal/portfolio, built using [Astro](https://astro.build), Tailwind and other utilities. It includes both component logic and build tooling for production, tests (unit and e2e), and performance checks such as Lighthouse and bundle budgets.

---

## 📦 Overview

The site is a static, SEO-friendly Astro project deploying to Vercel. Key features include:

- Fully typed with TypeScript
- Tailwind CSS for styling with a utility-first workflow
- Responsive layout and animations using `lenis` and custom helpers
- State management with `nanostores` for shared UI state
- Content-driven blog managed via `src/content` and `content.config.ts`
- Design tokens generated by Style Dictionary (CSS + TypeScript)
- Automated Lighthouse audits and bundle size budgets
- Unit tests with Vitest and DOM Testing Library
- End-to-end tests with Playwright

## 🚀 Getting Started (Local Setup)

### Prerequisites

- Node.js 20+ (tested on macOS Monterey/vent)
- `npm` (comes with Node) or `pnpm`/`yarn` (if you prefer)
- Git (to clone the repo)

### Installation

```bash
# clone & navigate
git clone https://github.com/jackcoventry/portfolio-2026.git
cd portfolio-2026

# install dependencies
npm install

# generate design tokens (required before dev/build)
npm run tokens
```

### Development

Start the development server with hot reload:

```bash
npm run dev
```

The site is available at `http://localhost:4321` by default. Use `--host` if you need network access.

### Building

Production build (runs token generation automatically):

```bash
npm run build
```

Preview the build locally:

```bash
npm run preview
```

### Utility Scripts

- `npm run lint` / `npm run lint:fix` – ESLint checks
- `npm run prettier` – format code using Prettier
- `npm run typecheck` – run `astro check` (TypeScript type validation)
- `npm run test` / `npm run test:coverage` – run unit tests with Vitest
- `npm run test:e2e` – run Playwright end‑to‑end tests
- `npm run lh:local` / `npm run lh:ci` – Lighthouse audits
- `npm run bundle:budget` – enforce JS/CSS size budgets
- `npm run verify` – the fast everyday validation pass (format, lint, typecheck, unit tests, build, bundle, route and link checks)
- `npm run verify:full` – the slower high-confidence pass for browser behavior and performance (`test:e2e` + Lighthouse)

> Husky is configured via `prepare` script for pre‑commit hooks. `lint-staged` ensures staged files are formatted and linted.

### Verification Workflow

- Use `npm run verify` during normal development. It is intended to stay fast and deterministic enough to run often.
- Use `npm run verify:full` before pushing, merging, or deploying when you want confidence in the real rendered experience as well as Lighthouse thresholds.
- `verify` intentionally does not include Lighthouse or browser E2E checks, because those are slower and more environment-sensitive. Keeping them separate makes the normal validation loop cheaper, which makes it more likely to be run consistently.

## 🗂 Project Structure

```
/
├── config/                # build & tooling scripts (style-dictionary, etc.)
├── public/                # static files, favicon, manifest
├── src/
│   ├── assets/            # images, svgs, fonts used by components
│   ├── components/        # reusable Astro/TS components + tests
│   ├── consts/            # shared constants
│   ├── content/           # Markdown/MDX or JSON content for pages/blog
│   ├── data/              # JSON data pulled in at build time
│   ├── layouts/           # common page layouts (Layout.astro)
│   ├── pages/             # route handlers (Astro pages)
│   ├── scripts/           # node utilities (Lighthouse, bundle budget)
│   ├── store/             # nanostores state modules
│   ├── styles/            # global styles, tokens, utils
│   └── utils/             # helper functions and shared TS modules
├── test-results/          # output from CI tests
├── tsconfig.json
├── package.json
└── vitest.config.ts       # unit test config
```

> Many overlapping helper files under `src/utils` mirror configuration in `config/` for the browser vs Node environment.

## 📝 Developer Notes

- **Content updates:** add Markdown/MDX files under `src/content/blog` or `pages`. `content.config.ts` controls collection parsing.
- **Design tokens:** modify JSON in `design-tokens/` and rerun `npm run tokens` to emit updated CSS/TS.
- **Linting and verification:** commit hooks run `prettier` and `eslint`. Use `npm run verify` as the normal local gate, and `npm run verify:full` for the final browser/performance pass before shipping.
- **Adding components:** place `.astro` files in `src/components`, accompanying tests in same directory using `.test.ts`.
- **Routing:** dynamic routes use `[...slug].astro` for slug handling.
- **Build environment:** environment variables are defined via Vercel dashboard; local `.env` is ignored (see `.gitignore`).

## ✅ Testing

- **Unit tests** live alongside components and utils. Run `npm run test` or `npm run test:ui` to open interactive UI.
- **E2E tests** are in `src/e2e/*`. Playwright config is in `playwright.config.ts`.

Results are output to `test-results/` and used by CI pipelines.
